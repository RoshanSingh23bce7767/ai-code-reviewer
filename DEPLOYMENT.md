# Production Deployment Guide

This repository is ready for a split production deployment:

- Backend API on Render using `render.yaml`
- Frontend on Vercel using `vercel.json`
- MongoDB through MongoDB Atlas or another managed MongoDB provider
- Optional Redis through Render Redis, Upstash, or another managed Redis provider

Recommended free public names:

- Backend: `codeinsight-ai-api.onrender.com`
- Frontend: `codeinsight-ai.vercel.app`

Final availability depends on whether those names are still free in Render and Vercel.

## 1. Required environment variables

Set these before running the stack in production:

```bash
export NODE_ENV=production
export PORT=10000
export MONGODB_URI="mongodb+srv://..."
export REDIS_URL="rediss://..."
export JWT_SECRET="$(openssl rand -hex 32)"
export CORS_ORIGIN="https://codeinsight-ai.vercel.app"
export CLIENT_URL="https://codeinsight-ai.vercel.app"
export SMTP_HOST="smtp.example.com"
export SMTP_PORT="587"
export SMTP_USER="your-smtp-user"
export SMTP_PASS="your-smtp-password"
export SMTP_FROM="no-reply@example.com"
export GEMINI_API_KEY="your-google-ai-studio-key"
export GEMINI_MODEL="gemini-2.5-flash-lite"
```

> Use a real secret manager or CI/CD secret store in production instead of hardcoding values. The app refuses to start in production when `JWT_SECRET` is missing, weak, or shorter than 32 characters.

Set these in Vercel for the frontend:

```bash
VITE_API_URL=https://codeinsight-ai-api.onrender.com/api/v1
VITE_APP_NAME=CodeInsight AI
VITE_AI_MODEL=gemini-2.5-flash-lite
```

## 2. Deploy backend to Render

1. Push the repository to GitHub.
2. In Render, create a new Blueprint or Web Service from the GitHub repository.
3. If using Blueprint, Render reads `render.yaml`.
4. If creating a Web Service manually, use:

```bash
Build command: npm ci --prefix server && npm run build --prefix server
Start command: npm run start --prefix server
Health check path: /health
Root directory: .
Runtime: Node
```

Set the backend environment variables in Render. `JWT_SECRET` can be generated by Render when using `render.yaml`.

After deployment, verify:

```bash
curl https://codeinsight-ai-api.onrender.com/health
curl https://codeinsight-ai-api.onrender.com/api
```

## 3. Deploy frontend to Vercel

1. Import the GitHub repository in Vercel.
2. Vercel reads `vercel.json`.
3. If configuring manually, use:

```bash
Framework preset: Vite
Install command: npm ci --prefix client
Build command: npm run build --prefix client
Output directory: client/dist
```

Set `VITE_API_URL` to the deployed Render API URL with `/api/v1` at the end.

After deployment, verify that registration, login, dashboard loading, and a code review request work against the Render backend.

## 4. Build and run with Docker

From the project root:

```bash
docker compose up --build -d
```

To stop the stack:

```bash
docker compose down
```

To view logs:

```bash
docker compose logs -f backend
```

## 5. Health checks

The backend exposes:

- /health
- /ready

The frontend container exposes:

- /health

Example:

```bash
curl http://localhost:5000/health
```

## 6. Production notes

- Render and Vercel provide HTTPS by default.
- Configure the proxy to forward the original client IP and set `trust proxy` correctly.
- Do not expose MongoDB or Redis to the public internet. Use managed services with credentials and IP/access controls.
- Keep `CORS_ORIGIN` and `CLIENT_URL` exactly aligned with the final Vercel frontend URL.
- Keep `VITE_API_URL` exactly aligned with the final Render backend URL plus `/api/v1`.
- Keep the database and Redis services backed up.
- Rotate secrets regularly.
- Monitor logs, `/health`, `/ready`, and `/metrics` in your hosting platform.
- Use alerts for failed health checks, elevated error rates, and backup failures.
